# RV1126B 部署全流程指南

目前你已经完成了 AI 开发中最核心的闭环测试（成功使用 demo跑通了 NPU 并输出 `out.png`）。接下来是将**你自己的目标检测代码（YOLOv5 的 EIoU/改进变体）**部署到 RV1126B 开发板上的标准工作流。

这里涉及到三套不同的「机器环境」：
1. **Windows 宿主机**：作为你写代码、看文档、跑纯 PyTorch 训练的主力平台。
2. **Ubuntu 虚拟机 / WSL**：作为模型转换（编译 RKNN）和 C/C++ 交叉编译的构建工厂。
3. **RV1126B 开发板**：只作为推理执行端。

不要把这三个环境的作用混淆。所有的源代码都留在电脑上，上传到板子上的只有“编译后的结果”和“运行素材”。

---

## 阶段一：Windows 宿主机上的工作（训练与准备）

在 Windows 原生环境下，你的主要工作是：

1. **训练模型**：在当前的 `bishe` 仓库中，运行 `train_ablation.py`，得到你最终改良版的目标检测 PyTorch 权重文件（如 `outputs/ablation_study/.../best.pt`）。
2. **导出 ONNX**：使用 `yolov5/export.py` 将 `.pt` 模型转换为通用的 `.onnx` 格式。此时 ONNX 模型依然是在 Windows 下生成的。

**结论**：Windows环境负责产出 `.onnx` 权重。

---

## 阶段二：Ubuntu (VM / WSL) 上的工作（转换与编译）

这是**工作量最大**的环节。你需要把 Windows 下准备好的模型源文件，移动或映射到 Ubuntu 环境中处理。

### 工作 1：Python 模型转换脚本（手写与执行）
在 Ubuntu 的 `RKNN-Toolkit2` Python 虚拟环境（如你的 `toolkit2` 环境）中：
- 你需要手写（或修改官方的）一个几十行的 **Python 转换脚本**（例如 `convert_yolo.py`）。
- 该脚本通过 `rknn.load_onnx()` 加载你在 Windows 生成的 `.onnx`。
- 设置平台配置 `rknn.config(target_platform='rv1126b')`。
- 使用你的训练数据集拿几百张图做校准集，执行 `rknn.build(do_quantization=True)` 做量化。
- 最后执行 `rknn.export_rknn()`，导出专属的 `best.rknn` 文件。

### 工作 2：C/C++ 板端应用代码（手写）
转换好模型后，你需要在 Ubuntu 中修改或新写一份 C/C++ 推理应用代码。这份代码将是未来跑在板子上的核心程序。
你需要按照 RKNN C-API 的规范编写：
1. `rknn_init`：加载前面生成的 `best.rknn`。
2. **前处理**：读取图片，利用 OpenCV 调整到信源尺寸（如 640x640）。
3. `rknn_inputs_set` 和 `rknn_run`：给 NPU 喂数据并执行运算。
4. `rknn_outputs_get`。
5. **后处理**（关键）：解算 NPU 吐出的大量浮点数，进行 NMS（非极大值抑制），将结果还原成检测框的坐标。这一步你需要根据你在 Windows 下修改过的网络输出结构做对应调整！
6. 画框并保存为结果图片。

### 工作 3：交叉编译（生成二进制）
**不要把写好的 `.cpp` 代码发到板子上**，板子不负责编译。
- 使用你在 Ubuntu 下配好的 `aarch64-linux-gnu-gcc` 或 `CMake` 工具来编译你的 C/C++ 源码。
- 编译时需明确链接库，如 `librknnrt.so` 和 `opencv`。
- 最终生成一个可在 arm 架构运行的**二进制可执行文件**（例如取名叫 `detect_infer`）。

**结论**：Ubuntu环境吃进 `.onnx` 和 `.cpp`，吐出 `.rknn` 模型、`detect_infer` 可执行文件。

---

## 阶段三：RV1126B 开发板上的工作（运行测试）

通过 ADB 将电脑端（Ubuntu 或 Windows）生成好的文件打包推送到开发板特定目录下（例如 `/userdata/deploy/`）。你需要上传一个典型的“最小运行包”。

### 你需要上传的内容清单：
1. **你的二进制可执行文件**：`detect_infer`。
2. **底层依赖库 (`lib/`)**：你的程序所依赖的 `.so` 库（如 `librknnrt.so`、打包进来用的 `libopencv_*.so`）。
3. **模型文件**：在 Ubuntu 里转换好的 `best.rknn`。
4. **资源文件**：几张用来测试的 `.jpg`，以及标注类别的 `labels.txt` 等。

### 在开发板上执行：
1. 连板并进入终端：`adb shell`。
2. 声明当前依赖库的位置，告诉板子去哪里找基础库：
   ```bash
   export LD_LIBRARY_PATH=./lib
   ```
3. 运行你的程序跑推理：
   ```bash
   ./detect_infer ./best.rknn ./test_image.jpg
   ```
4. 如果跑通了，就在同目录下取出程序刚刚生成好的带有检测框的输出照片查看效果。

## 总结：你的开发路径边界线

| 机器环境 | 环境承载任务 | 产出物 | 哪些文件在此环节 |
| :--- | :--- | :--- | :--- |
| **Windows 宿主** | 训练模型、改模型结构、评估算法指标 | `.pt`，`.onnx` | `train.py`、`yolov5源码`、`.yaml`配置、大量数据集 |
| **Ubuntu (WSL)** | 将 ONNX 转成 RKNN、**手写 C/C++ 推理代码**、执行交叉编译 | `best.rknn`，二可执行文件 (`detect_infer`) | `convert.py`、RKNN-Toolkit2环境、`main.cpp`等C/C++源码、`CMakeLists.txt` |
| **RV1126B 开发板** | 加载资源并调度 NPU 硬件真正进行推理 | 打印耗时、生成带框的检测结果图 | `detect_infer` (二进制)、`best.rknn`、`.so` 动态库、测试图片 `.jpg` |

> **一句话真理：** 你手写的**任何一行源代码（包括 `.py` 或 `.cpp` 或者 `CMakeLists.txt`）以及庞大的开发环境仓库本身，统统不需要上传到开发板！** 开发板只接受并且只认识编译后的可执行文件和 `.rknn` 二进制模型。
