## 2. 为什么单独放在 cpp 子工程

1. 不打乱当前 Python 仓库结构
2. 训练与部署分层清晰
3. 后续 Ubuntu 构建更容易
4. 便于做成独立作品集

## 4. 模块职责细化

### 4.1 common

职责：提供全局基础类型和公共能力。

建议包含：

1. Detection
2. Track
3. FrameResult
4. AppError
5. ScopedTimer

重点要求：

1. 类型定义尽量稳定
2. 不在这里写业务逻辑
3. 保持低耦合

### 4.2 config

职责：把 YAML 配置读取成强类型对象。

建议包含：

1. AppConfig
2. DetectorConfig
3. TrackerConfig
4. ConfigLoader

重点要求：

1. 配置缺项立即报错
2. 默认值写在配置结构中
3. 统一处理路径字符串

### 4.3 detector

职责：执行模型推理和前后处理。

建议拆分：

1. Detector 抽象接口
2. OnnxDetector 实现
3. Preprocess
4. Postprocess

输入：

1. OpenCV 图像
2. DetectorConfig

输出：

1. Detection 列表

### 4.4 tracker

职责：维护轨迹状态并生成跟踪结果。

建议第一版优先实现：

1. 简化 SORT
2. 或 ByteTrack 风格跟踪器

建议拆分：

1. TrackState 与 TrackStatus
2. Tracker 接口
3. Matcher
4. KalmanFilter
5. ByteTrackTracker 或 SortTracker

### 4.5 io

职责：处理视频输入输出和结果写盘。

建议包含：

1. VideoReader
2. VideoWriter
3. ResultWriter
4. SummaryWriter

### 4.6 pipeline

职责：把 detector、tracker、io 串成完整流程。

建议负责：

1. 初始化各模块
2. 循环处理每一帧
3. 写性能日志
4. 汇总输出结果

### 4.7 utils

职责：提供小型工具函数。

建议包含：

1. 画框
2. 画轨迹
3. 字符串处理
4. 时间格式化

## 5. 应用程序划分建议

建议至少做三个入口：

1. detect_image
2. detect_video
3. track_video

这样你可以按难度递增推进：

1. 先验证模型与后处理
2. 再验证视频 I/O
3. 最后验证整条跟踪流水线

## 6. 第一版技术选型

推荐：

1. C++17
2. CMake
3. OpenCV
4. ONNX Runtime
5. yaml-cpp

第一版不建议强行上：

1. LibTorch
2. TensorRT
3. Qt
4. 多线程任务系统
