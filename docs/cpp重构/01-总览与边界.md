# C++ 重构总览与边界

## 1. 目标

这次 C++ 重构的目标不是把整个仓库一次性改写，而是在 3 周内做出一条可运行、可部署、可继续扩展的 C++ 主线。

你的真实目标应该定义为：

1. 用项目练 C++ 工程能力
2. 做出可在 Ubuntu 20.04 上运行的检测与跟踪程序
3. 保持当前 Python 训练和实验主线不被破坏

## 2. 当前仓库应如何切分

当前仓库可以分为两类内容：

1. 训练与实验编排
2. 推理与跟踪执行

对你当前最有价值的重构方式是：

1. Python 保留训练、消融实验、评估、绘图
2. C++ 负责推理、跟踪、视频处理、部署

## 3. 为什么不建议全量改写

如果你现在追求整个仓库一次性重写成 C++，你会同时踩进以下问题：

1. 深度学习训练框架迁移
2. YOLOv5 训练生态替换
3. 数据与评估脚本替换
4. 推理部署问题
5. Ubuntu 构建问题
6. C++ 基础不牢造成的实现压力

这会让你的 3 周完全失控。

## 4. 本轮重构边界

### 4.1 保留在 Python 的部分

1. 数据准备脚本
2. YOLOv5 训练
3. 消融实验训练
4. 检测评估脚本
5. 跟踪对比绘图脚本
6. ONNX 导出流程

### 4.2 迁移到 C++ 的部分

1. ONNX 模型加载
2. 图像预处理
3. 检测后处理
4. 视频读取与保存
5. 跟踪算法
6. 结果写出
7. 性能统计
8. Ubuntu 20.04 部署

### 4.3 三周内明确不做

1. 不重写训练框架
2. 不做完整检测指标评估复刻
3. 不做全部脚本参数兼容
4. 不做 DeepSORT 的 ReID 特征网络
5. 不做 TensorRT 或 RKNN 多后端并行支持
6. 不做 GUI
7. 不做复杂多线程优化

## 5. 三周后的最小成果定义

你应该至少完成：

1. 一个独立的 C++ 子工程
2. 一个单图检测程序
3. 一个单视频检测程序
4. 一个单视频跟踪程序
5. 一个 Ubuntu 20.04 可编译运行版本
6. 一份部署说明或运行说明

## 6. 这次重构最重要的原则

1. 先跑通，再优化
2. 先单模块，再联调
3. 先 CPU 跑通，再考虑 GPU
4. 先 ByteTrack 或简化 SORT，再考虑更重方案
5. 每一周都要留下可运行成果

## 7. 建议阅读顺序

1. 先看本文件，理解边界
2. 再看工程目录与模块职责文档
3. 然后直接按三周计划执行
4. 遇到不会的点，再结合 C++ 知识地图补当前所需内容
